# 2. Arquitectura de información

Navegación, matriz de permisos, flujos principales, convenciones UI globales.

---

## 1. Mapa de navegación

```
/login                              (sin autenticación)

/                                   V1 Resumen (home)
├── /bi                             V2 Business Intelligence
├── /mapa                           V3 Mapa
├── /bbdd                           V4 Base de datos
│      └── ?optica=<place_id>       V5 Detalle (drawer overlay)
│
├── /changelog                      V6 Changelog (solo admin)
│
└── /admin                          área administrativa (solo admin)
       ├── /admin/usuarios          A1
       ├── /admin/operaciones       A2
       ├── /admin/revision          A3
       ├── /admin/cadenas           A4
       └── /admin/logs              A5

/perfil                             datos personales y password
```

**Notas:**
- V5 NO es página independiente. Drawer lateral derecho overlay sobre cualquier vista principal. Persiste en URL como query param para compartir enlaces.
- Sin breadcrumbs. Jerarquía plana en máximo 2 niveles. El sidebar marca la sección activa.
- `/admin/*` con rol `user` redirige a `/` con toast de aviso.
- En el prototipo Claude Design hay un **toggle de rol** persistente (esquina superior derecha) para alternar admin/user en runtime y enseñar la diferencia a CPV en review.

## 3. Estructura global de la interfaz

```
┌──────────────────────────────────────────────────────┐
│ [Toggle rol admin/user]                  [Avatar ▼] │  topbar prototype-only (esq. der.)
├──────────────────────────────────────────────────────┤
│  ┌────┐  ┌────────────────────────────────────────┐  │
│  │ S  │  │            ÁREA PRINCIPAL              │  │
│  │ I  │  │   (contenido de la vista activa)       │  │
│  │ D  │  │                                        │  │
│  │ E  │  │                                        │  │
│  │ B  │  │                                        │  │
│  │ A  │  │                                        │  │
│  │ R  │  │                                        │  │
│  └────┘  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

**Sidebar (64px icon-rail)** — fijo izquierda, fondo `--card`, separado del área por `border-right: 1px solid --line`. Brand marks (CPV+Newno) arriba. Iconos de navegación con tooltip a la derecha. Avatar abajo (menú: perfil, cerrar sesión).

**Área principal** — scroll interno. Header de vista (eyebrow, título, acciones) + contenido. Sin footer fijo.

**Drawer lateral derecho** — overlay 480px (560px en ≥1600px). Reservado a V5. Esc/backdrop/X cierran.

**Modales** — overlay centrado. Confirmaciones destructivas y formularios cortos. Máx 480px.

**Toggle rol admin/user** — control persistente en esquina superior derecha. Solo en el prototipo (no irá en producción). Permite alternar contexto en runtime y mostrar/ocultar elementos según rol activo.

## 5. Flujos principales por rol

### 5.1 `user` - consulta de mercado
1. Login → V1 Resumen
2. Identifica oportunidad en "Provincias con mayor oportunidad" (V2) → click provincia
3. Navega a V4 con filtro de provincia preseleccionado
4. Filtra por "no clientes"
5. Click en óptica → drawer V5
6. Consulta, cierra
7. Descarga listado filtrado en CSV

### 5.2 `user` - consulta puntual desde mapa
1. Login → V3 Mapa
2. Busca por nombre/ciudad
3. Click marcador → popup
4. "Ver detalle" → drawer V5
5. Consulta y cierra

### 5.3 `admin` - sincronización + revisión
1. A2 Operaciones → estado última sync
2. "Sincronizar ahora" Outscraper → confirma en modal
3. Sync en background → toast de inicio + banner persistente
4. Al finalizar → badge en sidebar sobre icono admin
5. A3 Revisión → bandeja con N pendientes
6. Procesa cada uno (vincular / no vincular / corregir datos)
7. Bandeja vacía → vuelve a V1

### 5.4 `admin` - corrección manual de dato
1. V4 → busca óptica con dato erróneo
2. Click → drawer V5
3. Tab Google Maps o Tab CPV según fuente del dato erróneo → click en campo
4. Modal "Corregir dato" → introduce valor correcto + motivo opcional
5. Sistema inserta en `opticas_overrides` y aplica on top
6. V5 muestra el campo con badge "Corregido manualmente" + link al log
7. Toast de confirmación

### 5.5 `admin` - alta de usuario
1. A1 Usuarios → "Nuevo usuario"
2. Modal: nombre, email, rol
3. Sistema genera password temporal + envía email
4. Toast de confirmación

## 6. Convenciones UI globales

### 6.1 Cuándo usar cada overlay

| Patrón | Uso |
|---|---|
| Drawer derecho | Detalle de óptica (V5). Mantiene contexto del listado detrás. |
| Modal centrado | Confirmaciones destructivas, formularios cortos, "Corregir dato" en V5. |
| Toast (abajo-derecha) | Confirmaciones, notificaciones de sync, errores no bloqueantes. |
| Banner (top área principal) | Alertas persistentes ("Sincronización en curso"). |
| Inline | Validaciones de formulario, mensajes contextuales. |

### 6.2 Confirmaciones destructivas

Modal para acciones irreversibles: eliminar usuario, descartar registro en revisión, cancelar sync, eliminar cadena. Descripción del efecto, no solo "¿estás seguro?". Acción primaria con color semántico negativo si aplica.

### 6.3 Búsqueda y filtros

- V3 y V4: búsqueda textual + filtros.
- A3, A4, A5, V6: filtros superiores compactos.
- Filtros activos como chips eliminables encima del listado.
- Búsqueda con debounce 300ms, sin botón "Buscar".

### 6.4 Paginación

- V4, V6, A5: paginación numerada 20/página (50/página en V6, A5).
- A1, A4: sin paginación (volúmenes bajos).
- A3: scroll infinito (cientos de pendientes posibles).

### 6.5 Ordenación

- Columnas ordenables con icono. Una columna a la vez. Persistencia entre sesiones (localStorage).

### 6.6 Exportación

- Botón "Exportar CSV" en V4 (respeta filtros) y A2 (BBDD completa). Descarga inmediata.
- Naming: `opticas_iberia_YYYYMMDD_HHmm.csv` o `opticas_filtrado_YYYYMMDD_HHmm.csv`.

## 7. Estados globales

| Estado | Cuándo | Tratamiento |
|---|---|---|
| Loading inicial | Primera carga | Skeleton screens. Sin spinners genéricos. |
| Loading parcial | Cambio de filtro | Skeleton solo en el área que cambia. |
| Empty | Sin resultados, primer uso | Ilustración mínima + mensaje + acción. |
| Error | Fallo API, timeout | Mensaje claro + "Reintentar". Nunca pantalla en blanco. |
| Sin permisos | user accede a admin | Redirect + toast. |
| Sesión expirada | Token caducado | Redirect login + toast. |

En el prototipo: cada vista expone selector de estado en una toolbar de prototipo (no visible al cliente en review final) para alternar default/loading/empty/error.

## 9. Navegación cruzada

- V1 → V2: click en KPI navega a sección correspondiente de V2.
- V2 → V4: click en provincia/cadena/tipo navega a V4 con filtro preaplicado.
- V2 → V3: "Ver en mapa" desde sección territorial.
- V3 ↔ V4: toggle en header para alternar mapa/tabla con mismo conjunto filtrado.
- V4 → V5: click en fila abre drawer.
- V3 → V5: click marcador → popup → "Ver detalle" abre drawer.
- V5 → V6: desde "Cambios y overrides", link "Ver en changelog completo" → V6 filtrada por esa óptica.
- A3 → V5: click en registro de revisión abre V5 en modo lectura.

## 10. Convenciones de URL

Filtros, búsqueda, ordenación y paginación viven en URL como query params:
- `/bbdd?provincia=Madrid&tipo=cliente`
- `/bbdd?optica=<place_id>` o `/mapa?optica=<place_id>`
- `/bbdd?q=Visionlab`
- `/bbdd?orderBy=rating&order=desc`

Permite compartir enlaces, atrás/adelante del navegador, y refrescar manteniendo estado.

En el prototipo Claude Design (SPA con routing en cliente), aplica lo mismo: las URLs cambian al navegar.
