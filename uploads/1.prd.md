# 1. PRD - BBDD Maestra de Ópticas Iberia

**Cliente:** CooperVision Iberia
**Implementador:** Newno (newno.marketing)
**Versión:** 3.0 (modelo de datos relacional + diseño en Claude Design)
**Fecha:** Mayo 2026
**Estado:** Definición funcional cerrada · Lote 0 pendiente de ejecución

---

## 1. Resumen ejecutivo

Aplicación web interna para CooperVision Iberia que consolida, mantiene y explota una base de datos maestra de ópticas y centros de contactología de España y Portugal (~15.000-19.000 establecimientos). La plataforma combina dos fuentes (Outscraper / Google Maps + Salesforce CRM de CooperVision) en un modelo relacional donde las tablas se mantienen separadas y se vinculan por FK, ofrece visualización analítica y permite a un rol admin mantener la calidad del dato.

**Producto NO comercial.** No reemplaza herramientas de captación o gestión comercial existentes de CooperVision. Es un sistema de inteligencia de mercado y gestión del activo BBDD.

## 2. Modelo de datos (arquitectura)

Decisión arquitectónica clave: **dos fuentes, dos tablas, una FK opcional**. No hay consolidación destructiva: cada fuente conserva sus datos originales.

### 2.1 Tabla `opticas_google`

**PK:** `place_id` (string, identificador único e inmutable de Google Maps)
**Fuente:** Outscraper / Google Maps API
**Editable desde la app:** NO (read-only fuera de la sync)
**Volumen:** ~15.000-19.000 registros

Campos:

```
place_id, google_id, cid, name, name_for_emails, phone, website,
business_status, verified, description, about, address, street, city,
state, postal_code, country, country_code, latitude, longitude,
located_in, category, subtypes, query, rating, reviews,
reviews_per_score_1, reviews_per_score_5, reviews_link, photos_count,
working_hours_csv_compatible, booking_appointment_link, owner_title,
location_link
```

### 2.2 Tabla `opticas_cpv`

**PK:** `CODIGO` (string, código de cliente CooperVision)
**FK:** `place_id_fk` → `opticas_google.place_id` (NULLABLE, único)
**Fuente:** Salesforce CooperVision
**Editable desde la app:** NO (read-only fuera de la sync)
**Volumen:** ~3.000-4.000 registros (subset clientes)

Campos:

```
CODIGO, GRUPO, DIRECCION, LOCALIDAD, PROVINCIA, TEL, EMAIL,
DP, COM, TIPOLOGIA, SEGMENTACION, LOCATION_ACC, place_id_fk
```

### 2.3 Tabla `opticas_app_data`

**PK:** FK `place_id` → `opticas_google.place_id` (1:1)
**Fuente:** Generada y editada desde la app por admin
**Editable desde la app:** SÍ
**Volumen:** una fila por óptica activa

Campos:

```
place_id, show_campañas_core, show_campañas_miopia,
cadena_resuelta_id, fecha_alta_en_app, notas_internas
```

### 2.4 Tabla `opticas_overrides`

**PK:** `id` (autoincremental)
**Fuente:** Append-only log de correcciones manuales sobre cualquier campo de las otras tablas
**Editable desde la app:** SÍ (insert; no se borra ni edita)

Campos:

```
id, tabla_origen ('opticas_google' | 'opticas_cpv'),
registro_id (place_id o CODIGO según tabla_origen), campo, valor_nuevo,
valor_anterior, usuario_id, fecha
```

**Lógica de aplicación:** cuando la app lee un registro, aplica los overrides activos on top del valor original. Si llega una sync nueva con un valor que coincide con el del override, el override se considera "resuelto en origen" y queda inactivo (sigue en el log para auditoría). Si la sync trae un valor distinto al override, el override permanece (la corrección humana prevalece).

### 2.5 Tabla `cadenas`

**PK:** `id`

Campos:

```
id, nombre, keywords (array), dominios (array), pais ('ES' | 'PT')
```

Resolución de cadena automática en `opticas_app_data.cadena_resuelta_id` por matching de nombre + dominio contra `cadenas.keywords` y `cadenas.dominios`.

### 2.6 Propiedades emergentes

- **`is_client(place_id)`** → `TRUE` si existe `opticas_cpv` con `place_id_fk = place_id`. NO es una columna, se computa al leer.
- **Estado de matching (para A3 Revisión):**
  - `match_confirmado` — fila `opticas_cpv` con `place_id_fk` no nulo, vínculo establecido.
  - `nuevo` — fila `opticas_google` cuya `fecha_alta_en_app` es de la última sync.
  - `solo_cpv` — fila `opticas_cpv` con `place_id_fk` nulo y motivo `no_encontrado_en_google`.
  - `conflicto` — fila `opticas_cpv` candidata a varios `place_id` con confianza media (rango configurable).
  - `no_encontrado` — fila previa de `opticas_google` ausente en última sync (flag temporal en `opticas_app_data`).

## 3. Contexto y objetivo

CooperVision dispone hoy de información parcial de su red de clientes (código de cliente, grupo, datos básicos), pero carece de visión completa del mercado óptico ibérico, geolocalización precisa, métricas de reputación online y datos de presencia digital. Esta plataforma resuelve esa brecha:

- Convierte la BBDD de clientes en un mapa completo del mercado óptico ibérico.
- Mantiene el dato vivo con sincronizaciones recurrentes (Outscraper cada ~3 meses, Salesforce cada ~6 meses) y permite intervención manual del admin para correcciones puntuales con trazabilidad total.
- Da visibilidad analítica a marketing, ventas y dirección sobre cuota, penetración, reputación y oportunidades.
- Garantiza trazabilidad total mediante `opticas_overrides` (correcciones manuales) y logs de actividad de usuarios.

## 4. Usuarios y roles

### 4.1 Perfiles

| Perfil | Departamento | Necesidad principal |
|---|---|---|
| Dirección | CooperVision | Visión ejecutiva del mercado y de su penetración |
| Marketing | CooperVision / Newno | Análisis competitivo, presencia digital, oportunidades |
| Ventas | CooperVision | Visualización territorial, consulta de fichas |
| Data Steward | Newno / CooperVision | Mantenimiento del dato, resolución de conflictos, migraciones |

### 4.2 Roles del sistema

**`admin`** (Data Steward + responsables Newno + responsables CPV)
- Acceso total a todas las vistas
- Insert en `opticas_overrides` para correcciones manuales
- Update en `opticas_app_data` (show_campañas, notas)
- Gestión de usuarios, syncs, cadenas, revisión, logs
- Descarga de BBDD en CSV

**`user`** (Marketing / Ventas / Dirección)
- Acceso solo-lectura a las 4 vistas principales (Resumen, BI, Mapa, BBDD)
- Acceso a V5 Detalle solo-lectura
- Descarga de BBDD en CSV
- Visibilidad de datos: total por defecto (decisión pendiente con cliente, §10.1)

### 4.3 Volumen estimado

10-25 usuarios totales. Mayoritariamente `user`, 2-4 admins.

## 5. Alcance funcional (MVP)

### 5.1 Vistas principales

- **V1 · Resumen** — Dashboard ejecutivo, KPIs globales del mercado y presencia CPV.
- **V2 · Business Intelligence** — Análisis profundo, partners vs resto, segmentación, presencia digital, oportunidades.
- **V3 · Mapa** — Mapa interactivo Iberia, marcadores por óptica, filtros tipo/valoración/búsqueda.
- **V4 · Base de datos** — Tabla densa paginada con todas las ópticas, búsqueda, filtros, ordenación, exportación.

### 5.2 Vistas secundarias

- **V5 · Detalle de óptica** — Drawer lateral derecho con 5 pestañas (Google Maps, CooperVision, Cambios y overrides, App data, Estado y campañas). Edición admin.
- **V6 · Changelog** — Vista de auditoría de `opticas_overrides`. Solo admin.

### 5.3 Administración

- **A1 · Usuarios** — CRUD de usuarios.
- **A2 · Operaciones de datos** — Estado de sync, sync manual, configuración de frecuencia, historial, descarga CSV.
- **A3 · Revisión de datos** — Bandeja de matching (vincular o no vincular registros `opticas_cpv` con `opticas_google`). Estados `conflicto`, `solo_cpv`, `no_encontrado`.
- **A4 · Cadenas** — Gestión de la tabla `cadenas`.
- **A5 · Logs de actividad** — Registro de acciones de usuarios sobre el sistema.

### 5.4 Autenticación

Login email + contraseña. Recuperación por email. Sesión persistente con expiración configurable. Sin SSO ni 2FA en MVP.

### 5.5 Exportación

Descarga CSV desde A2 (admin, completa) y desde V4 (cualquier rol, con filtros aplicados).

## 8. Estados de datos relevantes para el diseño

### 8.1 Estados de óptica (`is_client` + estado de matching)

| Estado | Significado | Visibilidad |
|---|---|---|
| Cliente (`is_client = true`) | Tiene fila en `opticas_cpv` vinculada | Acento en mapa, badge en tabla |
| No cliente (`is_client = false`) | Sin fila en `opticas_cpv` vinculada | Gris en mapa, sin badge |
| `match_confirmado` | Vínculo establecido | Sin marca especial |
| `nuevo` | Detectado en última sync sin precedente | Badge "Nuevo" durante 30 días |
| `solo_cpv` | Fila Salesforce sin vínculo | Visible solo en A3 hasta resolver |
| `conflicto` | Matching ambiguo entre fuentes | Visible solo en A3 hasta resolver |
| `no_encontrado` | Registro previo desaparecido en última sync | Visible solo en A3 hasta resolver |

### 8.2 Estado de visibilidad en campañas

Dos toggles independientes en `opticas_app_data`: `show_campañas_core` y `show_campañas_miopia`. Editables admin en V5, filtrables en V4. Default `true` para clientes. Reservados para mapa público (Fase 2) pero gestionables ya en MVP.

### 8.3 Aplicación de overrides

Cuando la app lee una óptica, aplica los overrides activos del log on top. Visualmente en V5 los campos con override muestran badge "Corregido manualmente" + link al cambio en `opticas_overrides`.

## 9. A3 Revisión: nueva semántica

Cambio importante respecto a versiones previas del PRD: A3 ya NO consolida campos. Decide si dos registros (uno de `opticas_google` y uno de `opticas_cpv`) representan la misma óptica.

### 9.1 Botones en focus card de conflicto

| Botón | Acción técnica |
|---|---|
| Sí, es la misma | Crea/actualiza `opticas_cpv.place_id_fk = place_id_candidato`. Cambia estado a `match_confirmado`. |
| Son distintas | No crea FK. Marca el candidato propuesto como descartado. La fila CPV queda en `solo_cpv` hasta nueva sync. |
| Corregir datos | Abre panel de overrides. Permite registrar correcciones manuales sobre campos de cualquiera de las dos fuentes (insert en `opticas_overrides`). NO consolida. |

### 9.2 Estados especiales

- **`solo_cpv`** → admin puede: buscar manualmente el `place_id` correcto, marcar como cerrada, o esperar.
- **`no_encontrado`** → admin puede: mantener (Google la perdió por error), marcar como cerrada, investigar.

## 12. Métricas de éxito

1. **Adopción interna.** ≥80% usuarios objetivo acceden ≥1 vez/mes en los 3 primeros meses.
2. **Mantenimiento del dato.** Ciclo completo cada 3 meses sin intervención técnica externa.
3. **Calidad del dato.** Registros `conflicto` post-revisión <5% del total tras cada sync.
4. **Reducción tiempo de consulta.** Consulta puntual de óptica <30 segundos.
5. **Trazabilidad.** 100% de correcciones manuales registradas en `opticas_overrides`.

## 13. Glosario

- **BBDD maestra:** conjunto relacional de `opticas_google` + `opticas_cpv` + `opticas_app_data` + `opticas_overrides`.
- **Cliente / Partner:** óptica con vínculo `opticas_cpv.place_id_fk` válido.
- **Cadena:** grupo comercial. Resolución automática por keywords + dominio en tabla `cadenas`.
- **Matching:** decisión de admin (en A3) sobre si dos registros refieren al mismo establecimiento.
- **Override:** corrección manual de admin sobre un campo de las tablas read-only. Append-only en `opticas_overrides`.
- **place_id:** identificador único de Google Maps. PK de `opticas_google`, FK desde `opticas_cpv`.
- **show_campañas_core / show_campañas_miopia:** flags de consentimiento para mapa público (Fase 2).
